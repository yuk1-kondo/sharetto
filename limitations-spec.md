# Sharetto 制限事項・既知の制約 仕様書

本ドキュメントは、Sharetto MVP（WebRTC A/B 転送版）における**技術的制限・未対応事項・ユーザー向け注意**を定義する。

---

## 1. スコープ

| 項目 | 内容 |
|------|------|
| 対象バージョン | PR #3 以降（`cursor/webrtc-ab-visual-14e3` ブランチ） |
| 対象機能 | 直接接続（WebRTC）/ サーバー経由（Firebase RTDB）/ QR・参加コード接続 |
| 目的 | 開発・運用・サポート時の判断基準と、ユーザーへの説明テンプレート |

## 2. 転送方式の制限

### 2.1 直接接続（WebRTC P2P）— モード A

| 制限 | 詳細 | ユーザーへの表示 | 回避策 |
|------|------|------------------|--------|
| NAT 越え失敗 | STUN のみ使用。**TURN サーバー未実装**のため、企業 FW・キャリア NAT・対称 NAT 等で P2P が確立しない場合がある | 「直接接続に失敗しました」→ サーバー経由切替を促す | **サーバー経由モード**へ切替 |
| 接続タイムアウト | 15 秒以内に ICE/DataChannel が確立しない場合は失敗扱い | 同上 | 再試行 or サーバー経由 |
| 同時 peer 数 | PC が受信ハブ。理論上複数接続可だが、**ブラウザ・CPU・メモリに依存**（目安: 同時 5 端末までを推奨） | 特になし（性能劣化として体感） | セッション分割 |
| chunk サイズ | **256KB 固定**（iPhone Safari 互換優先） | 特になし | 将来 512KB 段階試行を検討 |
| DataChannel バッファ | 送信側バックプレッシャーあり。極端に大きいファイルは転送時間が長い | 進捗バーで表示 | サーバー経由 or ファイル分割 |
| ブラウザ対応 | WebRTC DataChannel 必須。IE 非対応 | 非対応ブラウザでは動作しない | Chrome / Safari / Firefox / Edge 最新版 |

### 2.2 サーバー経由（Firebase RTDB）— モード B

| 制限 | 詳細 | ユーザーへの表示 | 回避策 |
|------|------|------------------|--------|
| 1 ファイル上限 | **50MB**（クライアント + DB ルール） | 「最大 50MB」 | 分割 or P2P（大容量向け） |
| エンコード方式 | Base64 で RTDB に保存。**実サイズ比 ~133%** の転送量 | 特になし | P2P モード推奨 |
| 転送速度 | Firebase 経由のため P2P より**遅い** | 特になし | 直接接続モード |
| 無料枠 | 転送量・書き込み回数に依存。**大容量の連続利用はコスト増** | 特になし | 10 分 TTL・P2P 優先 |
| セッション合計 | コスト最適化プラン上 **200MB/セッション** を推奨上限（ハード enforce なし） | 特になし | 運用監視 |

### 2.3 モード切替ルール

```
デフォルト: 直接接続（p2p）
  ↓ 15 秒以内に確立できない / ユーザーが手動切替
フォールバック: サーバー経由（relay）
```

- QR URL: `upload.html?session=XXX&mode=p2p|relay`
- PC 側タブで事前にモード指定可能

## 3. セッション・データ保持

| 項目 | 仕様 |
|------|------|
| セッション TTL | **10 分**（PC タイマー表示 + RTDB ルール + Cloud Functions 削除） |
| 参加コード TTL | **10 分**（`codes/{pin}` + クライアント側検証） |
| データ永続化 | **なし**。期限後は自動削除 |
| 再開 | 期限切れ後は新 QR / 新参加コードが必要 |
| Google ログイン | **任意**。未ログインでも受信可能 |

## 4. 端末・ブラウザ別制限

### 4.1 iPhone / iPad（Safari）

| 制限 | 詳細 | UX 対応 |
|------|------|---------|
| バックグラウンド停止 | タブを裏に回す・画面オフで **転送が中断・大幅低速化** しやすい | 「画面を開いたままにしてください」警告（要 UI 追加検討） |
| HEIC 形式 | iPhone 標準写真。**Windows 標準ビューアで開けない** 場合がある | 送信時トースト警告（実装済み） |
| PWA | ホーム画面追加可だが **iOS Web Push / バックグラウンド転送は制限** | manifest 提供（実装済み） |
| WebRTC | 基本対応。DataChannel + 256KB chunk で安定性確保 | — |

### 4.2 Android（Chrome 等）

| 制限 | 詳細 |
|------|------|
| バックグラウンド | Safari より寛容だが、**長時間バックグラウンドは中断リスク**あり |
| ファイル選択 | `multiple` 対応（実装済み） |

### 4.3 PC（Windows / Mac）

| 制限 | 詳細 |
|------|------|
| 受信専用ハブ | 複数モバイルからの同時接続は PC リソースに依存 |
| ZIP 保存 | ブラウザダウンロード。**File System Access API 未使用**（将来拡張可） |
| Three.js 演出 | 低スペック GPU では FPS 低下の可能性。`prefers-reduced-motion` で非表示 |

## 5. セキュリティ・プライバシー

| 項目 | 現状 | 制限 |
|------|------|------|
| 認証 | セッション ID / 参加コードを知る者がアクセス可能 | **リンク・コードの秘匿が唯一の防御** |
| 暗号化 | WebRTC 確立後は DTLS。**シグナリングは RTDB 平文** | 中間者がシグナリングを見られる理論的可能性 |
| サーバー経由データ | Firebase に一時保存（10 分） | 完全ゼロ知識ではない |
| Google ログイン | 任意。PC 側の履歴管理用途 | 転送自体には不要 |

## 6. 機能スコープ外（未実装）

以下は **MVP 範囲外** とし、Issue / 将来 PR で対応する。

| 機能 | 理由 |
|------|------|
| TURN サーバー | 無料安定運用が困難。フォールバックで代替 |
| HEIC → JPEG 自動変換 | canvas 変換はメモリ・互換コスト大 |
| エンドツーエンド暗号（追加レイヤー） | シグナリング設計の見直しが必要 |
| 複数 PC が同時受信 | P2P 星型は 1 PC ハブ前提 |
| オフライン再送 | Service Worker は静的キャッシュのみ |
| 転送再開（レジューム） | chunk プロトコル拡張が必要 |
| 管理者ダッシュボード | コスト・運用フェーズで検討 |
| ネイティブアプリ | Web のみ |

## 7. ユーザー向け注意文（コピペ用）

### 7.1 共通

> シェアっとは一時的なファイル共有サービスです。データは **10 分後に自動削除** されます。重要なデータは必ずダウンロードしてください。

### 7.2 直接接続がうまくいかない場合

> 「直接接続」が失敗した場合は **「サーバー経由」** に切り替えてください。会社・学校のネットワークでは直接接続が使えないことがあります。

### 7.3 iPhone 利用時

> 転送中は **この画面を開いたまま** にしてください。iPhone の写真（HEIC）は Windows で開けない場合があります。

### 7.4 ファイルサイズ

> サーバー経由モードでは **1 ファイル最大 50MB** です。大きなファイルは直接接続モードをお試しください。

## 8. 運用・監視のしきい値（推奨）

| メトリクス | 警告 | 対応 |
|-----------|------|------|
| RTDB 転送量（日次） | 無料枠 80% 超 | P2P 利用促進、relay 制限検討 |
| P2P 失敗率 | 50% 超（同一ネットワーク除く） | TURN 導入を評価 |
| セッションあたりファイル数 | 20 超 | レート制限検討 |
| Cloud Functions 削除遅延 | 5 分超 | 関数ログ確認 |

| 優先度 | 項目 | 制限解除効果 |
